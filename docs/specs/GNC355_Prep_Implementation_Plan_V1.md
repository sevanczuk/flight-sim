## 4. Stream A — AMAPI Documentation

### A1. Crawler — URL discovery and fetch

**Task type:** code
**Deliverable:** Python crawler script at `scripts/amapi_crawler.py`; tracking SQLite DB at `assets/air_manager_api/crawl.sqlite3` (schema per D-03); operational log.

**Scope:**
- Initialize `crawl.sqlite3` if absent; schema per D-03 (`urls`, `edges`, `crawl_runs`)
- Import pre-existing mirror: one pass over `assets/air_manager_api/wiki.siminnovations.com/` inserting one `urls` row per HTML file (`source='pre-existing'`, `status='fetched'`, `fetched_at`=file mtime, `http_status=200`, `content_bytes`=file size, `content_sha256` computed, `local_path` relative). This happens inside a dedicated `crawl_runs` row.
- Parse every pre-existing HTML file for outbound `a[href]` links; normalize each link URL per D-03; upsert into `urls` as `source='pre-existing'` or `source='seed'` if it was in the seed file; populate `edges` (with anchor text).
- Read seed URLs from `assets/air_manager_api/air_manager_wiki_urls.txt`; normalize each; upsert into `urls` with `source='seed'` if not already present. URLs already present from the pre-existing import are updated to record that they were also in the seed (no source overwrite; `seed` rows that are also pre-existing keep `source='pre-existing'` since `fetched_at` is set).
- Start a new `crawl_runs` row; fetch URLs where `status='pending'` one at a time with ≥1 second delay; user agent from `assets/air_manager_api/user-agent.txt`
- On successful fetch: write HTML to `assets/air_manager_api/wiki.siminnovations.com/` matching the existing `index.php@title=...` filename convention; update `urls` row with `status='fetched'`, `fetched_at`, `http_status`, `content_bytes`, `content_sha256`, `local_path`, `last_fetched_run_id`; parse for outbound links; upsert discovered URLs into `urls` as `source='discovered'` if new; populate `edges`
- On failure: increment `attempts`; set `last_error`; set `next_retry_at` = now + exponential-backoff interval (60s, then 5 min, then 30 min); set `status='failed'` only after 3 attempts
- Queue policy for discovered URLs: only `status='pending'` if the title matches the API-page whitelist (§4.5); otherwise `status='out-of-scope'`. External (non-wiki) URLs marked `out-of-scope` with `category='external'` or `'youtube'`.
- Append human-readable events to `assets/air_manager_api/crawl_log.txt` for casual inspection; DB is the authoritative state

**Stop conditions:**
- Frontier empty (`SELECT COUNT(*) FROM urls WHERE status='pending'` = 0)
- Wall-clock exceeds 4 hours → pause, update `crawl_runs.notes`, report (triggers consideration of concurrent-fetch approach per D-02 #3)
- Manual interruption → resumable; re-running the crawler starts a new `crawl_runs` row and picks up `status='pending'` rows

**Safety:**
- Respect robots.txt if present
- Retry once on transient failures; log and move on after final failure
- Never DELETE or MODIFY files in the existing wiki mirror — only write new files
- Never read or transmit credential-like content
- DB writes wrapped in transactions; an interrupted crawler leaves the DB in a consistent state

**Estimated duration:** 45–60 min script development + 2–4 hours fetch (background) + 30 min verification.

**Completion signal:** `assets/air_manager_api/crawl_complete.md` with summary derived from the DB (URLs fetched count by source, discovered count, failed count, duration, out-of-scope count).

### A2. Parse — structured extraction from HTML

**Task type:** code
**Deliverable:** `scripts/amapi_parser.py`; structured JSON output at `docs/reference/amapi/_index.json`.

**Scope:**
- Read all `index.php@title=...` files in `assets/air_manager_api/wiki.siminnovations.com/`
- Filter to API-function pages (title matches whitelist); skip installation/manual/FAQ pages for this parse
- For each API page extract:
  - Function name (from page title)
  - Category (derived from prefix: `Xpl`, `Msfs`, `Hw`, `Si`, `Viewport`, `Img`, etc.)
  - Signature (arguments + types) from the wiki's syntax section
  - Description text
  - Parameter table (name, type, description)
  - Return value (type + description)
  - Code examples (as verbatim text blocks)
  - Availability flags (which sims this function applies to — inferred from category + page content)
  - Cross-references (links to other API pages)
  - Source URL (reconstructed from filename)
- Produce:
  - `docs/reference/amapi/_index.json` — canonical structured data for all functions
  - `docs/reference/amapi/_metadata.json` — parse metadata (generated timestamp, source file count, any parse warnings)

**Parse robustness:**
- Wiki HTML is machine-generated; structure should be consistent
- Unrecognized page structures logged to a warnings file, not silently dropped
- Each function record includes `parse_confidence: high|medium|low` based on how many expected fields were extracted

**Estimated duration:** 30–60 min script development + minutes to run.

## 5. Stream B — Instrument Sample Analysis

### B1. Rename — copy to working directory with safe names

**Task type:** code
**Deliverable:** `scripts/rename_instrument_samples.py`; `assets/instrument-samples-named/` with 45 human-readable directories; `docs/knowledge/instrument_samples_index.md` manifest.

**Scope:**
- Read each UUID directory's `info.xml`; extract `<aircraft>` and `<type>`
- Generate safe name: `{slugified_aircraft}_{slugified_type}_{first_8_chars_of_uuid}`
  - Example: `04a6aa5d-7aad-42e4-9ed7-ca313b0e2edb` (Cessna 172 ADF) → `cessna-172_adf_04a6aa5d`
- Slugify: lowercase, replace whitespace with `-`, strip non-`[a-z0-9_-]` chars
- If any collision despite UUID suffix: use 12-char UUID prefix instead; if still colliding, append `_2`, `_3`
- Copy (not move) each UUID directory to `assets/instrument-samples-named/{safe_name}/`
- Preserve all files: `info.xml`, `logic.lua`, `preview.png`, `lib/`, `resources/`
- Generate manifest `docs/knowledge/instrument_samples_index.md` with table: UUID | Safe name | Aircraft | Type | Author | Sim compatibility | Dimensions (pref W×H) | File size

**Safety:**
- Originals in `assets/instrument-samples/` NEVER modified (D-02 #5; CC safety rule)
- Copy operation is fully reversible (just delete the copy)
- Rename script is idempotent — re-running over existing output does not corrupt

**Estimated duration:** 30 min script development + minutes to run.

### B2. Subset selection — diverse 8–10 for initial analysis

**Task type:** docs-only (CD work)
**Deliverable:** List of 8–10 instruments selected for deep analysis, documented in `docs/knowledge/instrument_samples_analysis_subset.md`.

**Selection criteria:**
- Coverage of instrument categories: gauge (e.g., ADF, HSI), display (PFD, MFD), GPS (shape-similar to GNC 355), engine cluster, clock/timer, switch panel, annunciator
- Coverage of simulation backends: some X-Plane-only, some dual, some FS2020/FS2024
- Coverage of complexity: at least one simple (< 100 lines Lua), one medium (200–500), one complex (500+)
- One touchscreen or GPS-like if present (direct relevance to GNC 355)

Selection happens AFTER B1 produces the manifest, because the manifest's type/author/sim-compat/size data drives the selection.

**Estimated duration:** 30–60 min CD review.

---

### B3. Pattern analysis — deep read of subset

**Task type:** docs-only
**Deliverable:** `docs/knowledge/instrument_patterns.md` — catalog of observed patterns.

**Scope:**
- CC reads all `logic.lua` files in the selected subset
- Identifies patterns across: initialization idioms, property declarations, image/canvas usage, event handling, animation, variable subscription, state management, error handling, lib/ module organization, resource loading
- Categorizes each pattern: **confirmed** (3+ instruments use it consistently), **common** (2 instruments), **one-off** (1 instrument, but noteworthy)
- For each pattern: name, description, code example, instruments where it appears, critical review (what's good, what smells, portability notes)
- Flags anti-patterns: things that work but are likely to bite (magic numbers, globals, inconsistent naming, missing error handling)
- Produces overall assessment: what subset of patterns we'd adopt vs. improve-upon for our own instrument

**Estimated duration:** 2–3 hours CC.

## 6. Stream C — GNC 355 Functional Spec

### C1. PDF extraction

**Task type:** code
**Deliverable:** `scripts/pdf_extract.py`; raw extracted text at `assets/gnc355_pdf_extracted/text_by_page.json`; extracted images at `assets/gnc355_pdf_extracted/images/`; extraction report.

**Scope:**
- Primary: pdfplumber text extraction, page by page
- Image extraction via pdfplumber's `page.images` and image bytes
- Per-page output: page number, raw text, image list (filename, bbox, size), table list (if tables detected)
- Any page where text extraction returns empty or looks garbled → flag for OCR
- OCR fallback (Tesseract) applied only to flagged pages; OCR output clearly labeled as OCR in the JSON
- Extraction report summarizes: total pages, pages with text, pages that needed OCR, tables detected, images extracted

**Safety:**
- Read-only operation on the PDF; never modifies the source
- Output directory at `assets/gnc355_pdf_extracted/` — new, does not exist yet

**Estimated duration:** 30–60 min script development + minutes to run.

### C2. Functional spec draft — GNC 355 operational behavior

**Task type:** docs-only
**Deliverable:** `docs/specs/GNC355_Functional_Spec_V1.md` + `docs/reference/GNC355_Pilot_Guide_Reference.md`.

**Scope:**
- CC reads the extracted text JSON + accompanying images
- Identifies sections in the Pilot's Guide that pertain to the GNC 355 specifically (the guide covers 4 devices; 355 material is interspersed with device-agnostic and other-device content)
- Separates content into two buckets:
  - **Operational behavior** → `GNC355_Functional_Spec_V1.md`. Includes: physical layout and controls, screen pages (map, flight plan, procedures, nearest, waypoint, direct-to, settings, COM), button and knob behaviors, mode transitions, text entry, menus, state persistence, audio/alert behaviors, COM radio functions, messages and annunciators.
  - **Pilot technique / aeronautical guidance** → `GNC355_Pilot_Guide_Reference.md`. Narrative archive; not input to implementation.
- Functional spec includes a **family delta appendix** per D-02 #9: for each significant functional area, noting where the GPS 175, GNC 375, and GNX 375 differ from the 355 (or match it)
- Functional spec uses the V1 naming convention per the spec review workflow
- Images from the PDF that document screen layouts/button arrangements are embedded or referenced from the spec

**Spec structure outline:**
```
# GNC 355 Functional Spec — V1
1. Overview (what it is, what we're implementing)
2. Physical layout & controls (bezel buttons, knobs, touch zones)
3. Power-on, self-test, startup state
4. Display pages (overview + per-page section each)
5. Flight plan editing
6. Direct-to operation
7. Procedures (approaches, arrivals, departures — if in scope)
8. Nearest functions
9. Waypoint information pages
10. Settings / system pages
11. COM radio operation
12. Audio / alerts / annunciators
13. Messages
14. Persistent state (what survives a power cycle)
15. External I/O (if documented — dataref inputs, command outputs)
Appendix A: Family delta (GPS 175 / GNC 375 / GNX 375 differences)
Appendix B: Glossary / abbreviations
Appendix C: Extraction gaps (pages where extraction had issues; OCR'd content flagged for review)
```

**Estimated duration:** 2–3 hours CC.

## 9. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Crawl discovers 500+ URLs; fetch exceeds 4 hours | Medium | Medium | A1 has a 4-hour stop gate; triggers concurrent-fetch recommendation (per D-02 #3) |
| Wiki HTML structure varies; parser misses fields | Medium | Medium | A2 emits `parse_confidence` per record; A4 spot-check catches systematic issues |
| Vendor sample code is genuinely bad; "patterns" section becomes a "don't do this" list | Low | Low | Already expected (per D-02 #5); pattern catalog has explicit anti-pattern section |
| GNC 355 content in PDF is insufficient for full spec | Medium | High | C1 extraction report flags gaps; some content may need supplementing from Garmin's cockpit reference guide (not in hand) or device screen walkthroughs |
| PDF has scanned (non-text) content requiring heavy OCR | Low | Medium | Tesseract fallback built into C1; OCR'd sections flagged in spec Appendix C |
| Samples `config.sqlite3` references UUIDs; disrupting AM install | N/A | N/A | We don't touch originals (D-02 #5); config.sqlite3 in assets/ is a local copy |
| CLAUDE.md placeholders (System Environment, test command, etc.) block a CC task | Medium | Low | CC tasks in this plan don't rely on those placeholders; they'll be filled when implementation work starts |
